# Experiments

> Track hypotheses, tests, and results. The area learns by running small experiments.
> Orchestrator proposes experiments. CEO approves high-risk ones.

## When to Propose an Experiment

1. A KR is off-track and the root cause is unclear
2. Two valid approaches exist and data is needed to choose
3. A playbook change is proposed but unproven
4. An agent discovers an unexpected pattern worth validating

## Instructions for Orchestrator

1. Propose experiments when KRs are off-track or when a decision lacks data.
2. Keep experiments small and time-boxed (1-2 weeks max).
3. Only 1-2 active experiments at a time — more creates noise.
4. If the experiment involves external-facing changes (outreach templates, pricing), add to DECISION_QUEUE.md for CEO approval first.
5. Always document the result, even if the experiment "failed" — failure IS learning.
6. After completing an experiment, check if PLAYBOOKS.md or LEARNINGS.md should be updated.
